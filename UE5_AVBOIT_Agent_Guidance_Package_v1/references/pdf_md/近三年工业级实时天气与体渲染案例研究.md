核心案例分析

天空与宏观体积

Nubis³ 是这批资料里最接近“你真正想做的局部异构体”的一例，但它不是直接给沙暴做的，而是围绕
Guerrilla 的实时体素云系统发展出来的 shipping 级体积方案。官方 2023 Advances 说明，这套系统在不到 6
个月里做出了高细节、可沉浸式探索的 voxel-based clouds，既能作为传统的 volumetric skybox 支持 time-
of-day，又能支持近距离穿云飞行；为此它采用了 压缩 SDF 加速 ray march、流体模拟式云建模、稠密体素上
采样、光采样加速，以及 dark edges / inner glow 的近似照明。公开索引到的 slide 还直接给出 voxel cloud
ray marcher 的成本区间约为 2.2–4 ms。官方 talk 页还额外公开了 Nubis Voxel Cloud Pack
supplementary materials，在近三年工业 talk 里算非常少见。


对你的题目，Nubis³ 的价值不在于“把云改成沙子”，而在于它证明了：shipping 级异构体效果可以建立
在“体素密度场 + 跳空/压缩采样 + 局部照明近似”上，而不必依赖完整流体求解或纯离线体数据播放。 这对沙
暴尤其关键，因为沙暴最难的是“中景到近景之间的体量感”，而 Nubis³ 恰好提供了一个能在工业预算内成立
的 macro-volume 范式。它的短板也很清楚：云的外观、相函数和照明假设并不等于贴地沙尘或雪崩粉雪；如
果你在 UE 里想“完整移植 Nubis”，那几乎肯定是 renderer/source 级高难度工程。更现实的做法，是只借
它在 模块 A–B 里的几个核心思想：局部密度场、压缩跳空采样、近景密度稳定性、体素上采样与低带宽访问。

透明和光照一致地工作。 对沙暴来说，它告诉你风沙并不只是 fog density 的问题，还涉及植被、表面、色调和
GI 的统一反应；对雪崩来说，它提示你雪不是一团白雾而已，最终还会影响植被状态、地表覆盖与场景照明。
移植到 UE 的难点很大，因为这已经是 RTGI / scene representation / material baking 级别的系统；但作为
模块 D 的“光照与世界状态接口”，它极有价值，尤其适合你在论文或作品集里证明：你的天气系统不是孤立
VFX，而是能推动环境状态变化的渲染子系统。


Snowdrop 的两场 GDC 2024 分享，虽然不是“天气效果”标题，但在工程上极其相关。官方会场页写明，
Snowdrop 在 Avatar: Frontiers of Pandora 这类拥有 dynamic time of day and weather 的大世界里，已
经把 indirect diffuse illumination、reflections、long-distance shadows、audio queries 都纳入了 ray
tracing，而且这些效果在主机上同时支持 30 Hz 和 60 Hz，并且还能运行在没有原生 DXR 的 PC 硬件上。更重
要的是，公开 slide 片段透出了一些非常适合天气体效果借用的工程细节：它的 probe system 基于 RTXGI/
DDGI，但增加了 light cache、average light、deferred lighting、probe GBuffer + shadow cache；其中
average light of each probe is mainly used for fog。另一个公开片段又显示，在同时使用 screen-space
tracing 和 world-space tracing 时，有一个案例只需要让 29% 的 rays 进入真正的世界空间 RT，也就是大约
每 14 个像素只打一根 world-space ray。当然，Snowdrop 也公开了它的限制：例如 probe spacing 相关的漏
光、走廊方向对齐、薄墙和大室内空间问题。


这类资料对你的直接启发，是：天气体不一定非要自己“从零点亮自己”。对沙尘、粉雪、薄雾这类效果，很
多时候真正需要的是一个足够便宜、又能随场景变化的 fog/volume lighting cache。Snowdrop 这条线就非
常适合放进你的 模块 D，尤其是你想让局部沙暴或雪雾在一天内、天气切换时和环境光/天空色一起变时。它的
局限也很具体：这仍然是 lighting infrastructure，不是 weather generator；而探针系方案在紧凑室内、薄墙
和高对比几何附近会有明显约束。所以，它更像 Nubis-lite 或 UE Local Fog 的灯光搭档，而不是你主角算法
本身。


表面积雪与地形形变

Santa Monica Studio 的雪渲染 是近三年公开材料里，对“雪崩之后场景应该长什么样”最实用的一例。GDC
2023 官方会场页表述得很清楚：他们重写了 snow displacement solution，并比较了旧系统和新系统的限制与
改进。公开索引到的 slide 透露，旧系统的构成非常复杂：艺术流程需要一个用于 snow material 的 flat
rendering surface，再用 screen-space parallax 位移；还需要匹配的 projection surface 去把 top-down
信息投到屏幕，并在其下方额外制作 maximum / minimum meshes 来描述未扰动雪面。


新系统的关键则更像一个“围绕相机滚动的地表积累子系统”：他们维护了一个会随着相机移动而滚动的
persistent top-down render target，在其中渲染 meshes 和 VFX，表示“我们正在往雪里 carve 多深”；这
些 carve 可由任意 mesh 写入，常见是 capsule 和 sphere 等角色近似体，并通过 tailored shader 去编码与
parallax 算法匹配的数据。随后，他们又受 GPU-driven rendering 启发，将 mesh 自动拆分成更小块，并只
对必要部分使用 tessellation；公开 slide 直接说，这一步是把整套系统 ship 到 PS4 的主要因素。总结页又写
明，这套系统最终在多个平台上出货，支持跨平台质量缩放，并且 fully integrated into standard workflow，
同时命中了主机性能目标。


这套方案几乎不解决空中的粉雪体积，但它精准解决了雪崩题目里常常被忽略、却会在答辩和作品评审里非常
加分的一层：地表反馈。如果你的雪崩只有一团白雾，没有堆积、压痕、车辙、脚印、边缘细节和厚度变化，
那整体可信度会立刻下降。对 UE 而言，这个案例的主要价值不在于逐像素移植，而在于它给出了一个非常工业
的“表面层”设计：滚动 render target、局部 carve、选择性细分、跨平台缩放。因此我会把它放在你的 模
块 D 的 surface feedback 子层，让雪崩从“只有 airborne volume”变成“空中粉雪 + 地面积雪反应”的完
整系统。

透明与近景 VFX 合成

AVBOIT 是这份报告里最需要单独拎出来讲清楚的一例，因为它常常会被误解成“天气方案”，其实它更准确
的定位是：天气/VFX 的透明合成后端。Activision Research 2026 页面明确写到，这项工作讨论的是 Call of
Duty 渲染引擎在支持 active game releases 的同时，如何从传统透明路径转向 order-independent
transparency，并给出了实现细节、性能画像、优缺点和现有游戏内容下的实际应用。SIGGRAPH 2025 的官
方 slides 又明确指出，游戏里典型的 transparency 包括 glass、VFX meshes、VFX cards，并强调如果透明
物体顺序不对，alpha blend 的 order dependence 会造成闪烁和伪影。


从算法上看，AVBOIT 先建立在 Voxel-Based OIT 的基本框架之上：官方 slide 给出的 basic steps 是，先拟合
一条 slice distribution over depth 的曲线，清空 volumetric extinction buffer，在低分辨率下把透明物体
splat 到 extinction buffer 里，然后按视线把 extinction 积分到 3D texture，再绘制 opaque、以任意顺序绘制
transparency，并最终 resolve。AVBOIT 相比基础 VBOIT 的关键增量，是它针对“透明层在 Z 轴上往往是稀疏
的”这一事实，引入了 1D Depth Warp LUT：先根据 Z occupancy 做 prefix sum，再 compact 并记录可过滤
的区间边界，让 splatting 和 sampling 都能跳过空深度区间。官方 slide 明确强调，这种只在 单一维度 Z 上做
sparse/warp 的方式，比全面在 XY+Z 上做稀疏结构更容易构造和采样。官方图示里，这个机制表现为一个按
深度压缩的 LUT，用来复映射非空 slice 区间。


在工程配置上，AVBOIT 的测试框架也非常工业。官方结果页写到：所有方法都使用相同的 transparency
rendering pass 与 resolve pass，transmittance prepass 在 1/8 分辨率运行；AVBOIT 的 prepass 输出采用
8bit × 128 slices（外加 32bit × 128） 并配合 8k Depth Warp LUT。性能上，官方给出了几组非常实用的
数据：在 PS4 1080p 下，MBOIT / WBOIT / AVBOIT / No OIT 的总成本分别是 6.65 / 6.16 / 5.03 / 4.5 ms；在
PS5 4K 的 practical performance 对比中，No OIT 为 2.8 ms，而 AVBOIT RGB 为 3.11 ms。官方结论页把这
套方法概括为 “GOOD ENOUGH”：更接近 No OIT 的速度、稳定、少量内容返工、RGB transparency
practical、可扩展，并且为 OIT refraction / scattering / transparency DOF / motion blur 打开了门。官方
slides 里最值得直接看的两页，就是 Depth Warp LUT 结构图 和 Gen8 min-spec 性能页。


对于沙暴和雪崩，AVBOIT 的意义几乎是立刻成立的。你的近景里一定会有 沙粒 cards、雪粉 cards、烟雾薄
片、爆炸 dust、玻璃/湿表面后方的透明层；这些东西如果继续使用传统排序，做得越多，越容易闪、穿、错
层。因此，AVBOIT 非常适合被定义成你的 模块 C：近景透明 / 粒子稳定合成层。它的短板也同样现实：它不会
生成天气，不会模拟密度，不会提供风场；而在 UE 里认真落地，基本一定要动到 源代码级透明 pass /
resolve / 渲染图（RDG）。所以它是一个非常好的第二阶段或源码版路线，而不是你第一周就该押上的唯一主
线。


面向 UE 的系统映射
如果把前面的工业案例压缩成一个真正适合 UE 项目落地的结构，我建议把你的 Weather Volume
Enhancement Layer 定义成四个模块：模块 A 为密度/风场，模块 B 为局部体积渲染，模块 C 为粒子与透明合
成，模块 D 为天气状态、表面积累、光照缓存和引擎整合。 这实际上也是近三年 shipping 案例共同呈现出来
的分工方式：GT7 管远景天气基底，Nubis³ 管宏观异构体，Hogwarts/AC 管世界状态与积累，Santa Monica
管地表反馈，AVBOIT 管透明合成，Snowdrop/AC RTGI 管雾与动态天气的光照接口。
